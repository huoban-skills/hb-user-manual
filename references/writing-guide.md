# 使用手册写作规范

本文件定义了生成使用手册时的写作风格、文档结构和注意事项。所有规范从实际优化迭代中沉淀而来。

---

## 一、文档整体结构

先判断模块类型，再选择结构。不要把业务流程模块写成单表手册。

### 1. 基础资料 / 数据维护模块

适用于客户、供应商、产品/物料、仓库、单位等资料维护类模块。

```
# 模块名

模块场景介绍

---

## 建议维护顺序
（仅基础资料/数据维护模块使用）

## 表间关系总览
（仅基础资料/数据维护模块使用）

## 一、表A

一段大白话背景介绍

### 1.1 新增/维护

1. 步骤
2. 步骤
3. 最后一步融入结果

### 1.2 字段说明

**表A：**
| 字段 | 填写方式 | 是否必填 | 说明 |

### 1.3 注意事项

### 1.4 常见问题/典型场景

## 二、表B
...
```

### 2. 业务流程模块

适用于销售、采购、生产、发货、收款、退款、退货等流程类模块。

```
# 模块名

模块场景介绍

---

## 一、业务闭环A

一段大白话背景介绍

### 1.1 操作步骤

1. 步骤
2. 步骤
3. 最后一步融入结果

### 1.2 字段说明

**主单表：**
| 字段 | 填写方式 | 是否必填 | 说明 |

**明细表：**
| 字段 | 填写方式 | 是否必填 | 说明 |

### 1.3 注意事项

### 1.4 常见问题/典型场景

## 二、业务闭环B
...
```

### 3. 结构规则

**基础资料 / 数据维护模块：**
- 可以写"建议维护顺序"和"表间关系总览"。
- 按表的依赖顺序排列章节，被依赖的表排在前面，如单位表排在产品/物料表前面。

**业务流程模块：**
- 不写"建议维护顺序"和"表间关系总览"。
- 按业务闭环组织章节，如销售报价、销售订单、销售发货、销售收款、销售退款、销售退货。
- 主单据和明细表如果共同构成一个业务动作，不要拆成两个大章。
- 明细表只有在有独立业务含义时才单独成章；否则作为主业务动作下的小节。
- 入口不决定章节归属。按钮在订单表上，不代表内容一定放进销售订单；如果它本身是独立业务闭环，就单独成二级标题，并在步骤里说明从哪个表/按钮发起。
- 示例：销售报价应写成一个大章，里面包含报价单创建、报价明细填写、报价审批（如有）、转正式订单等操作。
- 示例：销售发货、销售收款、销售退款、销售退货如果是销售角色发起的闭环业务，应单独成章；不要因为按钮入口在订单表就揉进销售订单。

**所有模块通用：**
- 不属于本模块的场景不写，用一句引导语"详见对应手册"带过。
- 每个二级章节末尾都要有"注意事项"和"常见问题/典型场景"，不要只在文末放一个全局注意事项或全局典型场景。
- 二级标题下必须有场景介绍，暗线包含是什么、为什么、怎么做；但文字不要模板化，"怎么做"只做简要概括，详细步骤放下面三级标题。
- 不加"业务定位"标题，章节标题下直接写介绍文字。
- 三级标题必须编号，格式如 `### 3.1 创建销售订单`，每个二级章节内从 `x.1` 开始。
- 不加"操作场景"标题包裹；如果只有一个场景，直接写场景名加粗即可。
- 场景格式：一句话说场景 + 编号步骤 + 结果融入最后一步。
- 禁止"触发时机/操作步骤/操作结果"三段式，业务人员看不懂。
- 字段表不拆两组：不要拆"手动填写字段/系统自动计算字段"，合并成一个字段表，用 `填写方式` 区分 `手动填写` / `系统自动`。
- 关联字段在说明列标注 `→ 目标表名`。
- 关联附加字段（从关联表自动带过来的字段）在填写方式列写 `系统自动`，说明中标注"从XX表自动带入，仅供查看"。
- 流程类系统字段不写：不要写流程状态、流程发起人、流程进展、流程启动时间、流程结束时间等平台过程字段。
- 业务判断字段要写：如订单状态、发货状态、待发货数量、已发货数量、待收款总额等。

## 二、写作风格

### 面向业务人员，不面向产品经理

- 语言像跟同事解释一样，不要说明书腔调
- 禁止技术术语：validator、trigger、数据触发、item_trigger、automation、config、node、API 等
- "数据触发自动xxx" → 改为"系统会自动xxx"
- "validator 校验" → 改为"保存时系统会自动检查，xxx时会弹窗提示"

### 复杂概念必须配例子

- 每个大章节的背景介绍中，尽量用具体例子帮读者理解，但不要套固定句式
- 字段说明中遇到不直观的概念（如损耗率、转化率、比例），用 `>` 引用块给出具体例子
- 例子要贴近制造业实际场景（螺丝、桌子、五金、电镀等）
- 拿不准的例子标记 `[待确认]`

### 举例示范

好的背景介绍：
> 供应商是企业的"上游"——你从谁那里买原材料、谁帮你做委外加工，都在这里管理。后续创建采购单时要选供应商，物料表里也可以标记每种物料的默认供应商。

不好的背景介绍：
> 记录企业所有供应商信息，是采购、委外、付款、对账的数据源。

好的字段说明：
> | 来源 | 选项 | 是 | 采购 / 自制 / 委外。"采购"就是从外面买的原材料，"自制"是自己工厂生产的，"委外"是发给外面工厂加工的 |

不好的字段说明：
> | 来源 | 选项 | 是 | 采购 / 自制 / 委外，决定获取方式 |

## 三、场景归属原则

每个场景只写在它所属的模块手册里，不跨界：

- 供应商/客户的**建档** → 基础资料模块
- 供应商**对账、付款** → 应付管理模块
- 客户**对账、核销、财务确认** → 应收管理模块
- 销售角色从销售单据发起的**收款、退款、退货** → 可以归入销售管理，作为独立业务闭环章节；章节内说明后续对账、核销、财务确认详见应收管理
- **批量采购**按钮 → 看情况，如果入口在基础资料表上就写在基础资料
- 生产工单、派工 → 生产管理模块

当某张表上有不属于本模块的按钮/操作时，用引导语带过：
> 供应商对账、采购付款录入等操作属于「应付管理」模块，详见对应手册。

### 参考成熟 ERP 手册的写法

- 先讲业务场景，再讲操作流程。
- 复杂动作讲应用场景、操作步骤、注意事项。
- 字段说明服务业务，不做全量字段堆砌。
- 典型场景要贴近日常使用，不要为了覆盖功能而强行编场景。

---

## 四、标记约定

| 标记 | 含义 | 使用场景 |
|------|------|----------|
| `[待确认]` | 需要用户校正 | 不确定的业务逻辑、举例是否准确、字段含义推测 |
| `[待补充]` | 需要后续积累 | 常见问题章节、还没有内容的注意事项 |

---

## 五、关联表的处理

当多张表构成主-明细关系时（如工艺+工艺路线、产品BOM+BOM清单），在同一个大章节下用子标题组织：

```
## 七、工序与工艺

（整体介绍）

### 7.1 工序
### 7.2 工艺
### 7.3 工艺路线
```

介绍时要讲清楚表之间的关系，比如"工艺是目录，工艺路线是正文"。
